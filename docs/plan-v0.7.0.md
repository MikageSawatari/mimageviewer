### ⛔ Task 8 (アプリデータ保存場所の変更機能) — **取りやめ**

一度実装 (`ddcd628`) したが、スモークテストで `App::on_exit` のタイミング競合バグ (起動時 in-memory の古い settings で移動完了後の disk を上書きしてしまい pending_move が消えない) を発見し、方針ごと撤回 (`d9e70ce` で revert、`79bd18e` でマニュアル微修正)。

**撤回の理由**:

- 移動途中に失敗した場合 (容量不足・ドライブ切断・権限エラー)、旧位置・新位置・settings.json の 3 点が整合しない状態になると、ユーザーが自力でリカバリできない。アプリ自体が起動しなくなる失敗パターンが存在する。

- 複数インスタンス起動をアプリ自動化時はロックで阻害する必要があったが、手動手順ならその制約も不要。ViX と同様に複数インスタンス起動を許容したまま運用できる。

**対応**: `htdocs/mimageviewer/manual/settings.html` に `--data-dir` を使った手動移動手順を追記 (コピー→検証→ショートカット書き換え、ドライブレター変更時の注意)。実装コード・`data_root` / `pending_move` フィールド・`instance_lock` モジュール・bootstrap/effective 2 段解決 は全て削除。

**影響を受ける計画**: Task 14-16 (RAR/7z/LZH → ZIP 変換) の `archive_cache/` も同じ場所に落ちるが、容量を分けたいユーザーは手動 `--data-dir` を使えばよい。Task 8 は計画から削除。

## 背景と狙い

### v0.7.0 の核となるストーリー

- **レトロ PC コレクション × AI アップスケール**という独自体験を提供
  - PI / MAG / Q0 / PIC / MAKI 等の PC-98 時代の画像フォーマットを、AI アップスケール
    と組み合わせて現代モニター解像度で鑑賞できるのは mImageViewer だけ
  - 例: 16bitセンセーション公式配布の LZH → 中身の PI → 4x AI アップスケールで 2560×1600
- **高速性ブランドを崩さないアーカイブ対応**
  - RAR / 7z / LZH はネイティブ対応ではなく「クリックで ZIP 変換」方式で受ける
  - ZIP に変換すれば以降は ZIP と同じ最速閲覧体験になる
- **競合（特に NeeView）との機能差を埋める**
  - ZIP in ZIP、ルーペ機能、背景色テーマ

## 機能一覧

### 1. Susie 画像プラグイン対応（32bit ワーカー、複数プロセス並列）

#### スコープ

**対応する**:
- Susie 画像プラグイン (`.spi` 32bit のみ)
- 代表的なプラグインで閲覧可能になる形式:
  - **PI** (ifpi.spi) — PC-98 標準の 16/256 色画像
  - **MAG** (ifmag.spi) — PC-98 の 256 色画像
  - **Q0 / Q4** — PC-98 フルカラー/16色
  - **PIC / PIC2** — シャープ X68000 系
  - **MAKI / HG / XLD** — その他レトロ形式
  - **PSD / JPEG 2000 / WebP 追加オプション等**（既にネイティブ対応のものは
    Susie 経由を優先しない）

**対応しない**:
- **Susie 書庫プラグイン** (`ax*.spi`)
  - ZIP/RAR/7z/LZH は純 Rust でカバー済み、それ以外（GCA/DGCA/YZ1/ISH/HKI）は死語
  - 書庫プラグインの position ベース API は画像プラグインと別ライフサイクル
    （追加実装コストが合わない）
- **Susie 64bit 版 (`.sph`)**
  - 古いレトロ画像プラグインは 32bit のみ、差別化の本丸は 32bit 側
  - TORO 系モダンプラグインの 64bit 版 (.sph) は mIV の既存ネイティブ対応と重複するため
    独自価値が低い

#### ワーカー構成（PDFium と同型）

```
mimageviewer.exe (64bit, メイン)
  ├─ mimageviewer-susie32.exe (32bit worker #1)
  ├─ mimageviewer-susie32.exe (32bit worker #2)
  └─ mimageviewer-susie32.exe (32bit worker #3)
```

- **ビルドターゲット**: `cargo build --target=i686-pc-windows-msvc --bin mimageviewer-susie32`
- ワーカーは GUI 依存を持たない細いバイナリ（`windows` クレート + 標準ライブラリのみ）
- デフォルト 3 ワーカー（環境設定で可変）
- IPC プロトコル: stdin/stdout バイナリ、PDFium ワーカーと同型（`src/pdf_loader.rs` 参照）
- 各ワーカーは起動時に設定ディレクトリから `.spi` を全ロードして拡張子→プラグインの
  マッピングを作る

#### Susie 画像プラグイン API の呼び出し

ワーカー側で:

1. `LoadLibrary` で `.spi` をロード
2. `GetPluginInfo` でプラグイン情報取得（対応拡張子など）
3. **画像デコード時**:
   - `IsSupported(filename, buffer)` で対応確認
   - `GetPicture(filename)` で DIB を取得
   - DIB → BGRA に変換してメインプロセスに IPC で返送

#### プラグインの配置先

- 既定: `<app_data>\susie_plugins\` （下記「アプリデータ保存場所」に従う）
- 初回起動時にこのフォルダを自動作成、`README.txt` を配置（入手先案内）
- **設定ダイアログに「プラグインフォルダを開く」ボタン**
  - Explorer でフォルダを直接開く（`ShellExecute` 経由）
  - ユーザーがドラッグ & ドロップで `.spi` を追加したら、設定側の「再読み込み」ボタンで
    ワーカー全員をリスタートして取り込む

#### ConfigurationDlg は非対応

Susie プラグイン API のうち、プラグイン側が自前の設定ダイアログを表示する
`ConfigurationDlg` 関数は v0.7.0 では対応しない。理由:

- ワーカープロセス側に Win32 メッセージループ・HWND クロスプロセス渡し・
  モーダル強制・ハング耐性の実装が必要で、工数対効果が低い（追加 2〜3 日）
- レトロ画像系プラグイン（ifpi, ifmag, ifpic, ifxld4 等）は ConfigurationDlg を持たず、
  v0.7.0 の差別化ストーリー（retro 画像 × AI アップスケール）には一切影響しない
- モダンプラグイン（ifjpegt, iftwic 等）で設定が必要な場合も、Susie 本体 (Susie.exe)
  を別途インストールしてそちらで設定すれば、同じ `.spi` を mImageViewer が使う時にも
  設定は反映される（多くのプラグインは同階層 INI または HKCU レジストリに保存するため）

ユーザー要望が蓄積したら v0.8.0 以降で、一時 32bit プロセスを起動してメインウィンドウ
HWND を親に設定ダイアログを表示する方式で対応を検討する。

#### クラッシュ隔離

- Susie プラグインはメンテ停止が多く、不正な画像でクラッシュするものが珍しくない
- ワーカープロセスがクラッシュしても、メインプロセスには影響しない
- メインはクラッシュしたワーカーを自動再起動（現状の PDFium ワーカー管理と共通化）
- **これ自体が訴求ポイント**: 「不安定な Susie プラグインを安全に使える唯一の実装」

#### 既存アーキテクチャとの統合

- `src/thumb_loader.rs` の画像デコード分岐に Susie ルートを追加
  - 拡張子で分岐（`.mag`, `.pi`, `.q0` 等）
  - Susie プラグインが対応していればそちらへルーティング
- キャンセル規約は `async-architecture.md` に従う
- サムネイルキャッシュは既存 SQLite+WebP をそのまま利用（フォーマットに依らず WebP で
  キャッシュされる）
- AI アップスケール・補正プリセットなどの下流パイプラインも既存のものをそのまま利用

### 2. クリックで ZIP 変換（7z / LZH）

**RAR は対応対象外 (2026-04-18 決定)**。`unrar` crate が内部で使う RARLAB unRAR ソースの
ライセンス (use-restriction 付き、Fedora が non-free と判定) が MIT 相当にならないため
除外。純 Rust の RAR5 solid デコーダは現状なし。ユーザーには FAQ で「WinRAR 等で一度
ZIP か 7z に変換してから」と案内する。

#### 基本動作

1. 7z / LZH ファイルがフォルダにあると、専用アイコン+拡張子バッジでグリッド表示
2. クリックすると確認ダイアログ:

   ```
   このファイルを ZIP に変換して閲覧します。
   mImageViewer は ZIP での閲覧に最適化されているため、
   一度変換するとその後は高速に開けます。

   入力: manga_vol01.7z （245 MB）
   出力先: <app_data>\archive_cache\...
           （元ファイルはそのまま残ります）

   画像ファイル数: 152（変換後 約 260 MB の ZIP を生成）

   [ 変換して開く ]  [ キャンセル ]
   ```

3. 「変換して開く」で進捗ダイアログ表示（キャンセル可能）
4. 完了したら展開済み ZIP を通常の ZIP 同様に開いてグリッド表示
5. 2 回目以降はキャッシュから即座に開く

#### 抽出バックエンド（すべて純 Rust）

| 形式 | クレート | 備考 |
|---|---|---|
| **7z** | `sevenz-rust2` | solid 圧縮・LZMA2 対応、純 Rust、MIT/Apache-2.0 |
| **LZH** | `delharc` | 純 Rust、LH0/LH5/LH6/LH7 対応、MIT/Apache-2.0 |

32bit ワーカー経由にはしない（純 Rust で完結するので IPC 不要）。

**7z の API 特性**: `sevenz-rust2` はシーク可能な Read が必要なため、stream 的な読み出しではなく
ファイルまたは `Cursor<Vec<u8>>` 経由でアクセスする。巨大な 7z の場合は一時ディレクトリに
コピーしてから開く (mIV の既存パターンと同じ)。

#### 出力ファイルの配置

- **出力先は常にアプリデータ配下** (`<app_data>\archive_cache\<hash>\<original_name>.zip`)
  - hash はオリジナルファイルの絶対パス＋更新時刻からの SHA-256
  - 元ファイルフォルダには何も出力しない（ユーザーのフォルダを汚さない）
  - 「元隣保存」オプションは用意しない（条件分岐を減らしてシンプル化）
- SQLite で「オリジナル → キャッシュ ZIP パス」のマッピングを管理（`archive_cache.db`）
- オリジナルの更新時刻が変わったら古いキャッシュを破棄して再変換

システムドライブ (`C:`) の容量圧迫を避けたいユーザーのために、
下記「アプリデータ保存場所の変更機能」で保存場所を別ドライブに移せる。

#### 変換中のみ画像ファイル

- ZIP に格納する対象は画像ファイルのみ（mImageViewer が扱える拡張子）
- 音声・動画・スクリプト・テキストなどは破棄
- 変換後 ZIP はメタデータに「変換元: XXX」を記録（将来の再処理に備える）

#### ZIP 圧縮レベル

- **STORE モード**（無圧縮）推奨
  - 中身は既に JPEG/PNG 等で圧縮済み、再圧縮の効果は薄い
  - 個別ファイルの読み出しが最速になる
  - ディスク消費は増えるが、mIV のサムネイル表示速度を最優先

#### キャッシュ管理（手動運用）

**自動削除は行わない** — 古い変換が意図せず消えてユーザー体験が悪化するため。

代わりに、キャッシュ管理ダイアログで**手動管理**できるようにする：

- `src/ui_dialogs/cache_manager.rs` を拡張して「変換アーカイブキャッシュ」タブを追加
- 一覧表示する情報:
  - 元ファイルパス（存在しないファイルは赤字表示）
  - 元ファイルサイズ
  - 変換後 ZIP サイズ
  - 変換日時
  - 最終アクセス日時
  - 元ファイル数（画像のみ）
- 操作:
  - 個別選択削除（チェックボックス複数選択）
  - 元ファイルが既に存在しないエントリの一括削除
  - 合計使用容量の表示とクリア機能

サムネイルキャッシュ（WebP）と比べて 1 エントリあたり数百 MB 〜 GB オーダーになりうる
ため、ユーザーが容量状況を把握できる UI を重視する。

#### 重要: 同名ファイル処理との整合

**懸念**: mImageViewer には現在、同名フォルダ・同名 ZIP・同名動画+画像などの重複を
設定に応じて非表示にする機能がある（`src/ui_dialogs/duplicate_settings.rs`）。
過去に Ctrl+↑↓ の前後フォルダ移動でこの非表示ロジックに起因するバグが複数回発生した。

ZIP 変換機能を追加する際の注意:

1. **変換後の元ファイル表示**
   - 変換済み RAR/7z/LZH は、設定によってグリッドから非表示にするか残すかを選択可能に
     するべき（現行の同名処理設定と同じ思想）
   - デフォルトは「変換済みバッジを表示したまま残す」が安全
     （いきなり元が消えると「自分のファイルが消えた」と誤解される恐れ）

2. **非表示ロジックを流用する場合の拡張**
   - `src/grid_item.rs` の `GridItem` に変換済みアーカイブを表す variant を追加

### 3. アプリデータ保存場所の変更機能 — **取りやめ (2026-04-18)**

当初案では、環境設定 → 全般に「アプリデータの保存場所」項目を追加し、ファイル単位の
copy → verify → delete で別ドライブへ移動する機能を実装する予定だった。コード上は
`Settings::data_root` / `pending_move` + 再起動時に移動を実行する Named Mutex 付き
排他制御までひと通り実装し (`ddcd628`)、手元テストまで進めたが以下の問題で撤回
(`d9e70ce`)。

**撤回の決め手**:

- スモークテストで `App::on_exit` のタイミング競合バグを発見。起動時に in-memory へ
  取り込んだ古い `pending_move=Some` が、移動 worker が disk に書いた新しい
  `pending_move=None, data_root=Some(...)` を終了時 save で上書きし、次回起動で
  「移動先に既存ファイルあり」エラーに陥る。ユーザーが自力で settings.json と
  新旧 2 箇所のデータを整合させる必要があり、**アプリが起動すらできなくなる失敗パターン
  が発生する**。
- 容量不足・ドライブ切断・書き込み権限エラーなど、移動途中のトラブルはアプリでは
  ハンドリングしきれないケースが多く、どれも上記のような「3 点不整合 → ユーザー自力
  復旧困難」に陥りやすい。
- 既に `--data-dir` オプションで同等機能が提供されており、「アプリ終了 → 手動コピー →
  ショートカットに `--data-dir` 追加」の手順でユーザーが完全可視・可逆に移動できる。
  複数インスタンス起動の排他制御も不要になり、ViX と同じく複数インスタンス許容のまま
  運用できる。

**現状の対応**:

- `htdocs/mimageviewer/manual/settings.html` の「別ドライブへのデータ移動」節に
  手動手順を記載 (フォルダコピー → ショートカット書き換え → 動作確認 → 旧削除、
  ドライブレター変更時の注意)
- `settings.rs` / `data_dir.rs` / `app.rs` / `main.rs` 等は Phase A の状態に戻し、
  `src/data_move.rs` / `src/instance_lock.rs` / `src/ui_dialogs/data_move.rs` は削除済み
- Task 14-16 の `archive_cache/` も通常 `%APPDATA%` に落ちる。容量を分けたい場合は
  ユーザーが `--data-dir` を使う

### 4. ZIP in ZIP のフラット展開

現状、ZIP 内のサブフォルダは無視してフラットに展開している。これをネストされた ZIP まで
展開する。

- 外 ZIP の central directory から内 ZIP エントリを発見
- 内 ZIP のバイト列を一時メモリまたは temp ファイルに展開
- 展開済み内 ZIP を `zip_loader` に食わせて中身を列挙
- GridItem として他の画像と並列にフラット表示
- パスは `outer.zip/inner.zip/image.png` のような表記で内部管理

**設計メモ**: `src/zip_loader.rs` に再帰展開のロジックを追加。展開済みバッファは LRU
キャッシュで保持し、使用されなければ自動廃棄。ネスト深度は制限なし（大半は 1 段、
深い場合でも 3 段程度が現実的）。

### 5. ルーペ機能

通常フルスクリーン・見開き・消しゴムモードで使える局所拡大機能。

- **Shift ホールド**: 押している間だけルーペ表示（主用途）
- **M キーでトグル**: 長時間の精密作業向け

#### 修飾キー選定の経緯 (Ctrl ではなく Shift にした理由)

mIV 全体の修飾キー慣習は Ctrl が主流 (Ctrl+wheel ズーム、Ctrl+drag 回転、
Ctrl+↑↓ フォルダ移動、Ctrl+1-0 プリセット適用、Ctrl+A/F/S など)。一貫性の観点では
Ctrl が素直だが、以下の理由で Shift を採用:

- **Ctrl+wheel の既存操作と視覚的に干渉する**
  - ズーム操作のたびに Ctrl 押下でルーペが一瞬表示 → wheel でズーム → Ctrl 離すで
    ルーペ消失、というちらつきが起きる
  - ズーム中は細部を見たいケースが多いとはいえ、毎回ルーペが一瞬出るのは煩わしい
- **Shift の既存バインドは消しゴムモードの傾き調整のみ**で衝突しない
- **Photoshop/Krita の一時ツール切替パターン** (Space=ハンドツール、Alt=スポイト等) と
  同じ発想

Ctrl+drag (回転) とも同様の干渉があるが、回転よりズームのほうが頻度が高いため、
ちらつき回避を優先する。
- カーソル周辺 200×200px 相当を 4 倍拡大して別レイヤ描画（倍率は設定可）
- カーソルから少しオフセットした位置にポップアップ表示
- ESC または M で OFF
- フォーカス喪失時（Alt+Tab 等）は自動 OFF（`ctx.input(|i| i.focused)` で検知）
- 画像分析モード（Z キー）とは独立

**実装場所**: `src/ui_fullscreen.rs` の描画パイプラインに別レイヤのサンプラーを追加。

### 6. 背景色テーマ（UI テーマ）

プリセット 3 種類のみ提供。自由色選択は将来要望が出たら追加。

- **標準** (デフォルト): サムネイル白背景、フルスクリーン黒背景（現状）
- **ライト**: 全体白ベース
- **ダーク**: 全体黒ベース

**初回起動時の挙動**:

- Windows システムテーマを自動検出（`UISettings.GetColorValue(UIColorType.Background)`）
- ダークモード → ダークを自動選択、ライトモード → 標準
- 画面右下に非モーダルなトーストを表示（数秒で消える）
  - 「テーマを変更できます: 設定 → 表示」

**検証**:

- WCAG AA コントラスト検証のユニットテスト（前景/背景組合せで 4.5:1 以上を自動確認）
- `egui_kittest` でスナップショットテスト（各テーマでメイン画面・フルスクリーン・
  設定ダイアログ・メタデータパネル・画像分析・AI 処理中 UI を撮る）

### 7. 透過画像の背景色（画像単位の機能）

テーマとは独立した、透過 PNG / WebP / GIF を開いた時だけ有効な機能。

#### 挙動

- フルスクリーン・見開き表示中、**B キー** (Background) で透過背景を循環
  - テーマ既定 → 白 → 黒 → 市松 → テーマ既定 …
- 画面右上に現在の背景モードを示す小さなインジケータを表示（自動フェードアウト）
- 透過を含まない画像では B キーは無効（または視覚的フィードバックなし）

#### 状態の保存

- **保存しない**（フラッシュ状態）
  - 1 枚ごとの好みは永続化コストに見合わない
  - 画像を開き直すたびテーマ既定に戻る
  - 要望が多ければ後で rotation_db と同等の画像別永続化を検討

#### 市松模様のデザイン

- 16×16 ピクセルの中間グレー市松（Photoshop/GIMP 標準）
- DPI スケーリングに追従

## FAQ ページ

v0.7.0 リリース時に追加する（`htdocs/mimageviewer/manual/faq.html` を新設）。

### 7z / LZH アーカイブに対応していますか？

「開くだけで即サムネイル表示」の形では対応していません。これらの形式は内部構造上、
一覧を作るだけで書庫全体の展開が必要になってしまうためです。

代わりに、これらのファイルをクリックすると ZIP に変換して閲覧できる機能をご用意して
います（元ファイルは残ります）。変換は初回のみで、以降は ZIP と同じ高速な閲覧体験に
なります。

### RAR アーカイブには対応していますか？

RAR の展開には RARLAB 社の unRAR ソースコードが必要ですが、このコードには mImageViewer
のライセンスと整合しない利用制限があるため、mImageViewer には組み込んでいません。
お手数ですが、WinRAR や 7-Zip 等のツールで一度 ZIP か 7z に変換いただき、変換後の
ファイルを mImageViewer でお開きください（7z はそのまま開けます）。

### ゲーム独自のアーカイブ（xp3、arc、tlg、g00 等）には対応していますか？

ゲーム固有のアーカイブ形式・独自画像形式には対応していません。専用ツールの GARBro が
数百のゲームエンジンに対応しており、独自画像形式（TLG / G00 等）の PNG 変換まで
自動で行ってくれます。GARBro で抽出・変換した画像フォルダや ZIP は、そのまま
mImageViewer で高速に閲覧できます。

### 古い PC の画像形式（MAG、PI、Q0 等）には対応していますか？

Susie 画像プラグインを使って対応しています。対応させたい画像形式の Susie 画像プラグイン
（`.spi` ファイル）を設定ダイアログの「プラグインフォルダを開く」から開いたフォルダに
置くと、mImageViewer から認識されます。

AI アップスケール機能と組み合わせることで、640×400 クラスのレトロ CG を 2560×1600 の
現代モニター解像度で鑑賞できます。

### Susie プラグインの個別設定を変更したい場合は？

mImageViewer からはプラグイン個別の設定ダイアログを開けません。設定が必要な場合は、
Susie 本体 (Susie.exe) を別途インストールし、そちらでプラグインを登録・設定してください。
多くのプラグインは設定を INI ファイル（`.spi` と同じ階層）またはレジストリに保存する
ため、同じ `.spi` ファイルを mImageViewer のプラグインフォルダに置けば、設定値はそのまま
反映されます。

### マウスジェスチャーには対応していますか？

現時点では未対応です。要望に応じて検討します。

### ブラウザからの画像ドロップに対応していますか？

現時点では未対応です。要望に応じて検討します。
